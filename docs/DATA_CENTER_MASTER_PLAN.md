# HearMe2nite 데이터센터 설계 보고서

- 문서 버전: 1.0
- 작성일: 2026-07-19
- 적용 대상: HearMe2nite Beta 및 정식 서비스 준비 단계
- 목적: 불필요한 계정·Room·데이터의 안전한 정리, 오류 복구와 백업, 고객 문의 대응 체계 수립

## 1. 결론부터

HearMe2nite 데이터센터의 1차 목표를 단순한 “삭제 요청 처리 화면”에서 **데이터 수명주기 관리 센터**로 변경하는 방향이 맞다.

다만 다음 정책은 적용하지 않는 것을 권장한다.

> 두 사용자가 모두 2일 동안 접속하지 않으면 Room이나 계정을 자동 삭제한다.

2일은 주말, 여행, 기기 변경, 알림 미확인만으로도 쉽게 지나가는 기간이다. HearMe2nite에는 대화·사진·날짜별 기록처럼 한번 삭제되면 사용자에게 큰 손실이 되는 데이터가 있으므로, 짧은 미접속만으로 영구 삭제하면 운영 비용보다 사용자 신뢰 손실이 더 커진다.

권장 방식은 다음과 같다.

1. **관찰:** 미접속, 미완성 가입, 빈 Room, 과도한 Room 생성을 탐지한다.
2. **분류:** 정상·관찰 필요·휴면 후보·휴면·삭제 예정으로 상태를 나눈다.
3. **보호:** 먼저 앱 내부 휴면 상태로 전환하며 데이터는 유지한다.
4. **안내:** 사용자에게 이유와 복원 방법, 삭제 예정일을 알린다.
5. **유예:** 충분한 복원 기간을 둔다.
6. **삭제:** 조건 재검증, 백업 확인, 감사 로그 기록 후 서버에서 삭제한다.

베타 2주 동안에는 **자동 영구 삭제를 실행하지 않고 후보 탐지와 관리자 검토까지만 운영**하는 것이 안전하다.

## 2. 최신 데이터센터 운영 방식과 비교

최근 서비스들은 “미접속이면 즉시 삭제”보다 데이터의 전체 생명주기를 관리한다.

| 운영 원칙 | 일반적인 현대 서비스 | HearMe2nite 권장안 |
|---|---|---|
| 데이터 최소화 | 필요한 데이터만 수집 | 접속 상태와 정리 판정에 필요한 최소 신호만 저장 |
| 보관 기간 제한 | 목적이 끝난 데이터는 정리 | 빈 계정·빈 Room과 실제 기록 Room을 다른 기준으로 관리 |
| 단계적 조치 | 경고 → 제한 → 휴면 → 삭제 | 관찰 → 휴면 후보 → 휴면 → 삭제 예정 → 삭제 |
| 복구 가능성 | 소프트 삭제와 유예 기간 제공 | 앱 내부 휴면 및 보관함을 거쳐 복원 허용 |
| 자동화 안전장치 | 미리보기, 승인, 감사 로그 | 자동 판정과 실제 실행을 분리하고 관리자 승인 적용 |
| 백업 | 정기 백업과 복구 훈련 | Realtime Database 일일 백업, 30일 보관, 복구 훈련 |
| 악용 방지 | 가입 속도 제한, 봇 판정, 사용량 제한 | 이메일 인증, App Check, Room 생성 한도, 이상 생성 탐지 |

개인정보는 목적에 필요한 범위로 최소 수집하고 필요 이상 오래 보관하지 않는 것이 기본 원칙이다. EU 집행위원회도 데이터 최소화와 보관 기간 제한을 명시하고 있으며, 한국 개인정보 보호법 제21조 역시 보유 목적이 끝난 개인정보는 지체 없이 파기하도록 규정한다. 다만 파기는 복구 불가능하게 해야 하므로, “운영상 편리하다”는 이유만으로 짧은 기간 후 삭제해서는 안 된다.

## 3. 핵심 용어를 사용자도 이해할 수 있게 정의

- **최근 활동:** 단순히 화면을 열어본 것만이 아니라 로그인, Room 생성·입장, 기록 저장, 대화 작성 등 실제 사용을 뜻한다.
- **빈 계정:** 가입했지만 이메일 인증, Room 참여, 기록 생성이 모두 없는 계정이다.
- **빈 Room:** 상대방이 없고 대화·기록·사진·미션 등 의미 있는 데이터가 없는 Room이다.
- **휴면:** 데이터는 남아 있지만 평상시 목록에서 분리된 상태다. 사용자가 다시 로그인하면 복원 안내를 받을 수 있다.
- **삭제 예정:** 사용자 안내와 유예 기간이 시작된 상태다. 예정일 전에는 취소할 수 있다.
- **영구 삭제:** 데이터와 인증 계정을 복구할 수 없게 제거한 최종 상태다.

## 4. 가장 중요한 설계 판단

### 4.1 Firebase 계정 비활성화와 앱 휴면을 구분한다

Firebase Authentication의 `disabled` 상태는 로그인을 막는다. 따라서 “로그인하면 다시 활성화”하려는 계정을 Firebase Auth에서 바로 비활성화하면 사용자가 로그인할 수 없어 스스로 복원할 수 없다.

HearMe2nite에서는 먼저 다음처럼 앱 데이터에 휴면 상태를 둔다.

```text
users/{uid}/lifecycle/status = active | watch | dormant | deletion_scheduled
users/{uid}/lifecycle/lastMeaningfulActivityAt
users/{uid}/lifecycle/statusChangedAt
users/{uid}/lifecycle/reasonCode
users/{uid}/lifecycle/deletionScheduledAt
```

사용자는 Firebase 로그인까지는 정상적으로 하고, 앱이 `dormant` 상태를 확인하면 “계속 사용하기” 버튼으로 즉시 복원한다. Firebase Auth의 `disabled`는 스팸, 공격, 약관 위반처럼 로그인을 막아야 하는 보안 조치에만 사용한다.

### 4.2 접속 시간 하나로 삭제하지 않는다

판정은 최소한 다음 신호를 함께 사용한다.

- Firebase Auth 생성일과 마지막 로그인 시각
- 이메일 인증 여부
- 앱 내부 `lastMeaningfulActivityAt`
- Room 참여 수와 Room 멤버 수
- 대화·기록·사진·미션 등 의미 있는 데이터 존재 여부
- 짧은 시간 내 계정 또는 Room 생성 횟수
- 삭제·보존 예외 표시 여부

`presence.lastSeen`은 브라우저 종료나 네트워크 문제로 부정확할 수 있으므로 참고 신호로만 사용한다.

### 4.3 자동 판정과 실제 삭제를 분리한다

예약 작업은 후보만 만든다. 실제 삭제 작업은 별도 큐에서 조건을 다시 검사한다.

```text
탐지 작업 → 후보 목록 → 관리자/정책 승인 → 사용자 안내 → 유예 기간
→ 실행 직전 재검증 → 백업 확인 → 삭제 → 감사 로그
```

이 구조는 예약 함수가 중복 실행되거나 도중에 실패해도 같은 데이터가 잘못 삭제되는 것을 막는다.

## 5. 베타 2주 운영안

베타 중에는 기준을 검증하는 것이 우선이며 영구 삭제 자동화를 켜지 않는다.

### 5.1 베타 후보 기준

| 분류 | 후보 조건 | 베타 중 조치 |
|---|---|---|
| 둘러보기 계정 | 가입 후 3일 이상, 이메일 미인증, Room 없음, 의미 있는 활동 없음 | 관찰 목록 표시 |
| 미사용 계정 | 가입 후 7일 이상, Room 없음, 의미 있는 활동 없음 | 관리자 검토 후보 |
| 빈 Room | 생성 후 3일 이상, 1명 이하, 의미 있는 데이터 없음 | 빈 Room 후보 표시 |
| 과다 생성 | 한 사용자가 24시간 내 Room을 3개 이상 생성 | 추가 생성 제한 후보 |
| 휴면 Room | 두 멤버 모두 7일 이상 의미 있는 활동 없음 | 휴면 후보만 표시, 삭제 금지 |
| 보존 Room | 대화·기록·사진 중 하나라도 존재 | 자동 삭제 대상에서 제외 |

### 5.2 베타 관리자 화면

기존 `데이터 요청` 메뉴의 우선순위를 낮추고 `데이터 수명주기` 메뉴를 중심으로 구성한다.

1. **운영 요약**
   - 전체 계정, 실제 사용 계정, 둘러보기 계정, 휴면 후보
   - 전체 Room, 빈 Room, 1인 Room, 2인 Room, 기록 보유 Room
2. **정리 후보**
   - 후보가 된 이유와 사용된 판정 신호
   - 마지막 로그인과 마지막 의미 있는 활동을 분리 표시
   - “제외”, “계속 관찰”, “휴면 예약” 조치
3. **생성 이상 탐지**
   - 사용자별 하루 계정·Room 생성 횟수
   - 동일 사용자의 반복 생성과 사용하지 않은 Room 수
4. **실행 미리보기**
   - 실제로 변경될 경로, Auth 계정, Storage 파일 수
   - 지금은 실행 버튼을 잠그고 예상 결과만 제공
5. **정책 설정**
   - 판정 기간과 한도
   - 베타/운영 모드
   - 자동 실행 여부는 기본 OFF

### 5.3 2주 동안 수집할 지표

- 가입 후 첫 Room 생성까지 걸린 시간
- Room 생성 후 두 번째 사용자 입장까지 걸린 시간
- 가입만 하고 아무 행동도 하지 않은 비율
- 1인 Room과 2인 Room의 비율
- Room당 실제 기록·대화·사진 존재율
- 1일, 3일, 7일 재방문율
- 사용자별 Room 생성 횟수 분포
- 휴면 후보 중 다시 돌아온 비율

이 지표를 본 뒤 정식 서비스의 기간을 결정한다. 숫자를 보기 전에 2일 삭제 기준을 확정하면 정상 사용자를 정리 대상으로 잘못 분류할 가능성이 높다.

## 6. 정식 운영 권장 정책

### 6.1 계정 정책

| 상태 | 권장 기준 | 조치 |
|---|---|---|
| 둘러보기 | 가입 후 7일, 미인증, Room·활동 없음 | 사용 안내 1회, 후보 표시 |
| 미사용 후보 | 30일 동안 로그인·활동·Room 없음 | 휴면 예정 안내 |
| 앱 휴면 | 안내 후 14일 동안 반응 없음 | 앱 내부 휴면 처리, 로그인 복원 가능 |
| 삭제 예정 | 휴면 후 추가 30일 동안 복원 없음 | 삭제 예정일 안내 |
| 영구 삭제 | 예정일 도달 및 예외 없음 | 서버 삭제, 감사 로그 보존 |

베타 종료 후 실제 기간은 개인정보 처리방침, 서비스 성격, 사용자 반응을 검토해 확정한다.

### 6.2 Room 정책

Room은 계정보다 더 보수적으로 다룬다.

- **빈 1인 Room:** 생성 후 7일 동안 상대방 입장과 기록이 없으면 보관함으로 이동한다.
- **2인 무기록 Room:** 두 사용자 모두 30일 활동이 없을 때 휴면 후보가 된다.
- **기록 보유 Room:** 2일 또는 30일 미접속만으로 자동 삭제하지 않는다.
- **한쪽만 활동:** 활동 사용자가 있는 한 자동 휴면·삭제하지 않는다.
- **한쪽 계정 삭제:** 공동 기록의 권리를 먼저 판정하고 개인 데이터만 분리 삭제한다.
- **Room 전체 삭제:** 두 사용자 동의 또는 명확한 정책·공지와 긴 유예 기간이 필요하다.

### 6.3 무분별한 생성 방지

삭제보다 생성 단계의 통제가 비용과 위험을 더 많이 줄인다.

- 이메일 인증 전 Room 생성 금지
- 사용자당 활성 Room 수 제한
- 24시간 내 Room 생성 횟수 제한
- 빈 Room이 이미 있으면 새 Room 생성 전 기존 Room 사용 안내
- 초대코드 생성 횟수 제한과 만료 데이터 자동 정리
- App Check를 모니터링 모드로 먼저 적용한 뒤 차단 모드로 전환
- 계정·Room 생성 속도, 실패율, 동일 패턴을 감사 로그에 기록

OWASP는 대량 계정 생성을 자동화 위협으로 분류하며, 생성 속도 제한, 미완성·미사용 계정 모니터링, 신규·저사용 계정의 기능 제한을 권장한다. Firebase App Check는 앱이 아닌 곳에서 백엔드가 무단 호출되는 것을 줄이는 데 사용할 수 있다.

## 7. 1차 개발: 데이터 수명주기 센터

### 7.1 신규 데이터 구조 제안

```text
dataLifecycle/
  users/{uid}/
    status
    createdAt
    lastLoginAt
    lastMeaningfulActivityAt
    roomCount
    meaningfulDataCount
    reasonCodes
    reviewState
    noticeSentAt
    deletionScheduledAt
    protectedUntil

  rooms/{roomCode}/
    status
    createdAt
    memberCount
    activeMemberCount
    lastMeaningfulActivityAt
    hasMeaningfulData
    reasonCodes
    reviewState
    deletionScheduledAt
    protectedUntil

  policies/current/
    mode
    thresholds
    autoExecutionEnabled
    updatedBy
    updatedAt

  jobRuns/{runId}/
    jobType
    startedAt
    completedAt
    scannedCount
    candidateCount
    successCount
    failureCount

  actionQueue/{actionId}/
    targetType
    targetId
    action
    status
    snapshot
    approvedBy
    scheduledAt
```

원본 사용자 데이터에 분석 필드를 계속 섞기보다, 판정 결과를 `dataLifecycle` 아래에 분리하면 규칙과 감사 범위를 관리하기 쉽다.

### 7.2 의미 있는 활동 기록

서버에서 다음 행동이 성공했을 때만 갱신한다.

- Room 생성 또는 정상 입장
- 날짜 기록 저장
- 채팅 메시지 전송
- 사진 업로드
- 미션·루틴 생성 또는 완료
- 피드백 저장

단순 페이지 열기, 새로고침, 온라인 상태 변경은 의미 있는 활동으로 보지 않는다.

### 7.3 예약 작업

Firebase의 예약 함수 `onSchedule`을 사용해 하루 한 번 후보를 계산할 수 있다. 예약 작업은 재시도될 수 있다고 가정하고 동일 작업을 여러 번 실행해도 결과가 망가지지 않는 멱등 구조로 만든다.

- `scanLifecycleCandidates`: 매일 후보 계산
- `sendLifecycleNotices`: 안내 대상 생성
- `archiveDormantEntities`: 승인된 대상만 휴면 전환
- `executeScheduledDeletion`: 유예 종료·백업 확인·조건 재검증 후 삭제
- `verifyLifecycleIntegrity`: Auth, users, userRooms, roomMembers, rooms, Storage 불일치 검사

## 8. 2차 개발: 오류 복구와 백업

### 8.1 백업 목표

- Realtime Database 데이터와 Rules를 매일 자동 백업
- 기본 30일 보관
- 대규모 변경과 삭제 실행 전에 수동 스냅샷 생성
- Storage 사진은 객체 수명주기와 소프트 삭제 정책을 별도로 적용
- 백업 파일 접근 권한은 최소 관리자와 백업 서비스 계정으로 제한

Firebase Realtime Database는 Blaze 요금제에서 데이터와 보안 규칙의 일일 자동 백업을 Cloud Storage에 저장할 수 있고, 30일 수명주기 정책을 제공한다. Cloud Storage는 TTL, 저장 등급 변경, 이전 버전 보존, 소프트 삭제를 지원하지만 수명주기 규칙은 먼저 개발 데이터나 제한된 경로에서 시험해야 한다.

### 8.2 백업이 있어도 필요한 것

백업은 복구 기능 그 자체가 아니다. 다음 항목이 있어야 실제로 복구할 수 있다.

- 복구 담당자와 승인자
- 복구 대상과 우선순위
- 복구 전 현재 데이터 별도 보존
- 부분 복구와 전체 복구 절차
- 복구 후 무결성 검사
- 분기별 복구 훈련 기록

권장 목표:

- RPO(허용 가능한 데이터 손실): 최대 24시간, 중요 삭제 직전은 0에 가깝게
- RTO(복구 목표 시간): 베타 24시간, 정식 서비스 4시간 이내를 장기 목표로 설정

### 8.3 무결성 검사 항목

- Auth 계정은 있으나 `users/{uid}`가 없음
- `users/{uid}`는 있으나 Auth 계정이 없음
- `userRooms`와 `roomMembers` 연결 불일치
- `rooms`는 있으나 멤버가 없음
- 멤버는 있으나 Room 데이터가 없음
- Storage 사진은 있으나 참조 기록이 없음
- 기록이 참조하는 사진 파일이 없음
- 삭제 작업은 완료됐으나 일부 경로가 남아 있음

## 9. 3차 개발: 문의·고객센터

### 9.1 사용자 화면

- 문의 유형: 사용 방법, 데이터 오류, 계정·Room, 결제(추후), 신고, 개선 제안
- 제목과 내용
- 관련 화면과 발생 시각
- 선택적 첨부 이미지
- “개인정보를 입력하지 마세요” 안내
- 접수·답변 대기·답변 완료 상태

### 9.2 데이터센터 화면

- 문의 목록, 담당자, 우선순위, 상태
- 사용자에게 보이는 답변과 내부 메모 분리
- 관련 UID·Room은 기본 마스킹
- 감사 로그와 처리 시간 기록
- 데이터 오류 문의는 복구 사건으로 연결

### 9.3 권장 구조

```text
supportTickets/{ticketId}/
  ownerUid
  category
  title
  message
  status
  priority
  createdAt
  updatedAt

supportReplies/{ticketId}/{replyId}/
  visibility: user | internal
  message
  authorUid
  createdAt
```

사용자는 본인 문의만 읽고 쓸 수 있으며, 관리자는 전체 문의를 처리한다. 내부 메모는 사용자에게 절대 노출되지 않는 별도 경로와 보안 규칙을 사용한다.

## 10. 보안상 반드시 보완할 부분

1. **클라이언트에서 삭제 금지:** 영구 삭제와 Auth 관리는 Cloud Functions의 Admin SDK에서만 실행한다.
2. **최소 권한:** 관리자 읽기, 검토, 승인, 실행 권한을 장기적으로 분리한다.
3. **이중 승인:** Room 전체 삭제와 대량 삭제는 두 단계 승인을 적용한다.
4. **감사 로그 불변성:** 일반 관리자가 과거 감사 로그를 수정·삭제하지 못하게 한다.
5. **민감정보 마스킹:** 목록에서 이메일, UID, Room Code 전체 노출을 최소화하고 상세 열람을 기록한다.
6. **대량 작업 제한:** 한 실행당 처리 건수 제한과 중단 스위치를 둔다.
7. **Dry Run 기본값:** 모든 정리 작업은 미리보기부터 시작한다.
8. **보호 표시:** 분쟁, 복구 중, 사용자 문의 중인 데이터에는 `protectedUntil`을 설정한다.
9. **비밀 관리:** 서비스 계정 키를 저장소나 브라우저 코드에 넣지 않는다.
10. **App Check 단계 적용:** 먼저 모니터링하고 정상 요청 영향 확인 후 강제한다.

## 11. 사용자에게 보여줄 쉬운 정책 문안

### 베타 기간 안내문

> HearMe2nite는 현재 베타 테스트 중입니다. 서비스 안정성과 개인정보 보호를 위해 장기간 사용하지 않은 빈 계정과 빈 공간을 확인할 수 있습니다. 대화, 기록, 사진이 있는 공간은 짧은 미접속만으로 자동 삭제하지 않습니다. 정리가 필요한 경우 먼저 안내하며, 안내된 기간 안에 다시 로그인하면 계속 사용할 수 있습니다.

### 휴면 예정 안내문

> 최근 사용 기록이 없어 계정이 휴면 상태로 전환될 예정입니다. 계속 사용하려면 안내된 날짜 전까지 로그인해 주세요. 휴면 상태가 되어도 바로 삭제되지 않으며, 로그인 후 계속 사용하기를 선택하면 복원할 수 있습니다.

### 삭제 예정 안내문

> 휴면 상태가 오래 유지되어 데이터 삭제가 예정되었습니다. 삭제 예정일 전까지 로그인하면 삭제를 취소할 수 있습니다. 삭제가 완료되면 계정과 개인 데이터는 복구할 수 없습니다. 공동 공간의 기록은 상대방의 권리를 확인한 뒤 별도 기준에 따라 처리됩니다.

### 빈 Room 안내문

> 이 공간에는 아직 상대방이나 작성된 기록이 없습니다. 일정 기간 사용하지 않으면 공간 목록에서 보관 처리될 수 있습니다. 보관된 공간은 삭제 예정일 전까지 다시 활성화할 수 있습니다.

## 12. 다음 개발 순서

### STEP A12.1 — 관찰 전용 데이터 수명주기 센터

- 의미 있는 활동 기준 확정
- 사용자·Room 수명주기 스냅샷 생성

### STEP A12.2 — 정책과 휴면 미리보기

- 정책 설정 화면
- Dry Run 결과와 영향 경로 표시
- 보호·제외·계속 관찰 기능
- 사용자 안내 문안과 예정일 미리보기

#### 구현 상태 (2026-07-19)

- 관리자 화면에서 후보 기간과 Room 생성 점검 기준을 임시 조정할 수 있다.
- 변경한 정책으로 후보 수, 사용자 안내 예정일, 휴면 유예 종료일을 다시 계산한다.
- 각 후보에서 관련 `users`, `userRooms`, `rooms`, `roomMembers` 영향 경로를 펼쳐 볼 수 있다.
- 보호·제외·계속 관찰 선택은 현재 브라우저 메모리에만 유지되며 새로고침하면 초기화된다.
- 데이터베이스 저장, 실제 휴면 전환, 자동 삭제는 계속 비활성화되어 있다.

### STEP A12.3 — 앱 내부 휴면과 복원

- `dormant` 상태 적용
- 로그인 후 원클릭 복원
- Room 보관함
- 자동 삭제는 계속 OFF

#### 구현 상태 (2026-07-19)

- 관리자가 수명주기 후보 화면에서 사용자 계정을 수동으로 휴면 전환할 수 있다.
- 휴면 전환 시 계정과 연결된 Room Code만 보관 목록으로 기록하며 Room과 기록 원본은 변경하지 않는다.
- 휴면 사용자가 로그인하면 일반 앱 대신 데이터 보존 안내와 원클릭 복원 화면을 표시한다.
- 복원하면 계정 상태가 즉시 `active`로 돌아가며 기존 Room 연결과 기록을 그대로 불러온다.
- 휴면 전환은 관리자 확인이 필요하고 감사 로그를 남긴다. 자동 휴면과 자동 삭제는 계속 비활성화되어 있다.

### STEP A13 — 승인형 삭제 엔진

- 서버 전용 실행
- 백업 확인
- 실행 직전 재검증
- 이중 승인과 감사 로그
- 소량 대상부터 단계적 활성화

#### STEP A13.1 구현 상태 (2026-07-19)

- 승인된 계정 삭제와 Room 전체 삭제 요청만 서버 사전점검 대기열에 등록한다.
- 서버가 요청 상태, 사용자·Room 존재 여부, 연결 Room 수, 멤버 수, 백업 검증 상태와 상대방 동의를 다시 확인한다.
- 첫 승인자와 다른 관리자만 2차 승인을 기록할 수 있다.
- 검증된 백업이 없으면 2차 승인을 차단하며 모든 사전점검과 승인 시도를 감사 로그에 남긴다.
- `executionEnabled`는 서버에서 항상 `false`로 고정되어 있다. 이번 단계에는 영구 삭제 함수가 존재하지 않는다.
- 기존의 승인된 Room 연결 해제 기능은 공동 기록을 보존하는 제한 기능으로 그대로 유지한다.

#### STEP A13.2 구현 상태 (2026-07-19)

- 관리자 스텝, 이름, 캐시 키와 릴리스 날짜를 `admin/admin-release.js` 한 곳에서 관리한다.
- 왼쪽 메뉴 배지, 상단 배지, 감사 로그, 릴리스와 시스템 화면이 같은 관리자 버전을 표시한다.
- 감사 로그 화면에서 삭제 승인 대기열, 백업 확인 여부와 실행 잠금 상태를 함께 확인한다.
- 시스템 화면에서 승인 대기열 수, 검증된 백업 수와 실행 활성 건수를 별도 안전 지표로 표시한다.
- 실행 활성 건수의 정상 기준은 항상 0건이며 영구 삭제는 계속 잠금 상태다.

## 13. 최종 권고

다음 스텝은 삭제 기능이 아니라 **STEP A12.1 관찰 전용 데이터 수명주기 센터**로 시작한다.

베타 2주 동안 실제 데이터를 삭제하지 않고 “어떤 계정과 Room이 왜 후보가 되는지”를 측정한다. 이후 측정 결과와 사용자 반응을 바탕으로 휴면 기간과 삭제 유예 기간을 결정한다. 이 방식은 최신 데이터 최소화·보관 기간 제한 원칙을 따르면서도, 관계 기록형 앱에 필요한 복구 가능성과 사용자 신뢰를 지킬 수 있다.
