# HearMe2nite 관계 종료·회복 정책

## 목적

관계 종료는 Room 삭제가 아니다. 기존 기록은 보존하되 Dom과 Sub 양쪽의 읽기·쓰기를 함께 잠그고, 두 사용자가 다시 동의했을 때만 동일한 Room을 회복한다.

## 상태

- `active`: 기존 Room 데이터를 읽고 쓸 수 있다.
- `ended`: 양쪽 모두 Room 데이터, 사진 메타데이터, 채팅, 코멘트, 알림을 앱에서 읽거나 쓸 수 없다.
- `recovery_pending`: 데이터 잠금은 유지되며 상대방의 회복 동의를 기다린다.

상태 필드가 없는 기존 Room은 호환성을 위해 `active`로 본다.

## 종료

- Room의 기존 owner와 partner만 종료할 수 있다.
- 실행 전에 양쪽 데이터 잠금 안내와 `관계 종료` 확인 문구 입력이 필요하다.
- 저장 또는 사진 작업이 진행 중이면 종료하지 않는다.
- 멤버십과 데이터는 삭제하지 않는다.
- 종료 즉시 데이터 리스너, 자동 저장, Presence 연결을 중단한다.

## 회복

- 기존 owner 또는 partner 한 명이 회복을 요청한다.
- 요청한 사용자는 스스로 승인할 수 없다.
- 반대쪽 사용자가 동의해야 `active`로 전환된다.
- 회복할 때 `generation`을 증가시켜 새로운 관계 활성 주기를 기록한다.
- 새로운 사람은 기존 관계 회복에 참여할 수 없다. 새로운 관계는 새 Room과 일반 초대를 사용한다.

## 보안 경계

- Realtime Database 규칙은 멤버십뿐 아니라 `rooms/{roomCode}/meta/relationship/status`가 `active`인지 확인한다.
- 종료 상태에서도 관계 상태 경로만 기존 멤버에게 읽기·전환이 허용된다.
- `ownerNotes`를 포함한 Room 기반 데이터에도 같은 활성 상태 조건을 적용한다.
- Storage 다운로드 URL을 이미 외부에 복사한 경우까지 즉시 회수하는 것은 별도 토큰 폐기 또는 서버 작업이 필요하다. 앱에서는 종료 후 Database가 잠기므로 사진 URL과 메타데이터를 다시 불러오지 않는다.

## 배포 원칙

앱 Hosting과 Realtime Database 규칙이 함께 적용되어야 보안 기능이 완성된다. 한쪽만 먼저 배포하지 않는다. Storage, Functions, 관리자 앱은 별도 승인 없이는 변경하거나 배포하지 않는다.
