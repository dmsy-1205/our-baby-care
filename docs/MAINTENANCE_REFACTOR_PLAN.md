# HearMe2nite 유지보수 정리 계획

## 목적

현재 테스트 앱의 기능과 화면을 기준선으로 보존하면서, 한 부분을 수정했을 때 이전 UI나 다른 환경 설정이 다시 나타나는 회귀를 줄인다. 관리자 앱, Database 규칙, Storage 규칙은 별도 승인 전까지 변경하지 않는다.

## 고정 원칙

- 새 기능보다 회귀 방지와 구조 정리를 우선한다.
- 테스트 앱에서만 변경·검수한 뒤 메인 앱으로 이동한다.
- 배포 기본 대상은 테스트 앱이다. 메인 앱은 `Production`을 명시한 경우에만 배포한다.
- Hosting만 배포하며 Database, Storage, Functions는 포함하지 않는다.
- 관리자 앱은 파일 지문으로 동결하며 명시적 관리자 작업 승인 없이는 기준을 갱신하지 않는다.
- 기능과 디자인을 동시에 크게 바꾸지 않는다.
- 한 단계마다 라이트·다크, 전체 테마, Dom·Sub, 모바일·Fold·태블릿·PC를 확인한다.

## 단계

### 1. 기준선과 배포 안전장치

- 버전과 서비스 워커 버전 일치 검사
- Firebase 테스트·메인 별칭 검사
- 테스트 우선 배포 검사
- 중복 DOM ID와 JavaScript 문법 검사
- `public` 화이트리스트 및 규칙 파일 제외 검사
- 거대 파일과 HTML 직접 이벤트가 더 증가하지 않도록 상한 고정

### 2. 공통 화면 규칙 추출

순서는 `토큰 → 공통 상단 바 → 버튼 → 카드 → 모달`이다. 기존 CSS를 즉시 삭제하지 않고 새 공통 파일로 옮긴 뒤 화면 비교가 끝난 규칙만 원본에서 제거한다.

예정 구조:

```text
css/
  tokens.css
  components/
    app-shell.css
    top-bar.css
    button.css
    card.css
    modal.css
  screens/
    home.css
    daily.css
    records.css
    settings.css
```

### 3. 저장과 날짜 전환 통일

- 모든 입력 상태를 `작성 중 / 저장 대기 / 저장 중 / 저장 완료 / 저장 실패`로 통일한다.
- 임시 사진은 계정·Room·날짜를 키로 구분한다.
- 날짜 변경, Room 변경, 로그아웃 전에 미저장 상태를 처리한다.

### 4. Dom/Sub 권한을 한곳에서 판정

- UI 숨김만 권한으로 사용하지 않는다.
- 읽기·작성·수정·삭제·존재 숨김을 각각 판정한다.
- 권한표와 실제 판정 함수가 어긋나면 자동 검수에서 실패하게 만든다.

### 5. 직접 이벤트와 전역 함수 축소

- 새 HTML `onclick`, `oninput`, `onchange` 추가를 금지한다.
- 화면별 이벤트 위임으로 조금씩 옮긴다.
- 기존 전역 함수는 사용처가 사라진 뒤에만 제거한다.

## 단계별 완료 조건

1. 자동 기준선 검사 통과
2. public 화이트리스트 빌드 통과
3. 테스트 앱 배포 후 핵심 시나리오 통과
4. 모바일·Fold·태블릿·PC에서 레이아웃 회귀 없음
5. 모든 테마의 라이트·다크 모드에서 잔여 색상 없음
6. 메인 앱 배포 전 사용자 승인
